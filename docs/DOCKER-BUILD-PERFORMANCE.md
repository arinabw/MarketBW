# Сборка Docker: время и что можно менять

## Логирование этапов по времени

В **`docker/Dockerfile`** перед/после основных шагов выводятся строки **`[build-time]`** с UTC-временем. В логе сборки сравните **START** и **END** одного этапа.

Удобно смотреть полный лог без сжатия прогресса:

```bash
cd docker
DOCKER_BUILDKIT=1 docker compose build --progress=plain 2>&1 | tee /tmp/marketbw-build.log
grep '\[build-time\]' /tmp/marketbw-build.log
```

Типичный порядок «тяжести» (при **холодной** сборке, без кэша слоёв):

| Этап | Почему долго |
|------|----------------|
| **frontend: npm ci/install** | Скачивание npm-пакетов фронта. |
| **frontend: vite build** | Сборка Vue (тысячи модулей). |
| **runtime: pip install** | Установка FastAPI/uvicorn (обычно быстро). |

После первого успешного билда **BuildKit-кэш npm** и закэшированные слои Docker ускоряют повторные сборки, если не использовать **`--no-cache`** без необходимости.

## Нужно ли менять стек

**Менять что-то имеет смысл**, если узкое место подтверждено логами и время критично.

### 1. Не собирать фронт на сервере

- В **CI**: `npm ci && npm run build` (в `frontend/`), пуш образа в registry.
- На VPS: **`docker compose pull && up -d`** — без Node на сервере.

### 2. Ускорить pip

- Зафиксировать версии в `backend/requirements.txt` и полагаться на кэш слоя Docker.

### 3. Раньше было два процесса (nginx + Node)

Сейчас **один** процесс **uvicorn** на порту **8000** — отдельный nginx в образе не используется.

---

## Кратко

1. Смотрите **`[build-time]`** в логе и при необходимости **`--progress=plain`**.  
2. **Первый** билд всегда самый долгий; дальше помогают кэш и **`frontend/package-lock.json` + `npm ci`**.  
3. Для «мгновенного» деплоя на сервере — **готовый образ из CI + pull**.

# Влияние многослойного Docker‑образа на работу сайта

В `docker/Dockerfile` используется **многоэтапная сборка**:

```Dockerfile
# Сборка: Node.js
FROM node:20-alpine AS base
...
FROM base AS builder
... npm run build

# Production: только nginx и статика, без Node
FROM nginx:alpine AS runner
COPY --from=builder /app/dist /usr/share/nginx/html
COPY docker/nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 3000
CMD ["nginx", "-g", "daemon off;"]
```

## Что происходит
1. **Этап `builder`** собирает Vue‑приложение с помощью Node.js и сохраняет готовую сборку в `/app/dist`.
2. **Этап `runner`** берёт только **статические файлы** из предыдущего этапа и помещает их в образ Nginx. Node.js в финальном образе **отсутствует**.

## Как это влияет на компоненты сайта
- **Клиентская часть** (Vue‑компоненты) полностью компилируется в статические HTML/CSS/JS файлы. После сборки они работают без необходимости Node.js – браузер просто получает готовый JavaScript‑bundle.
- **Серверная часть** (Node‑сервер) не используется в продакшене. Все запросы обслуживает Nginx, который отдает статические файлы и проксирует API‑запросы (если они настроены).
- **Плюсы**:
  - Меньший размер образа (только Nginx + статика).
  - Быстрее старт контейнера, меньше уязвимостей, так как Node.js не присутствует.
  - Упрощённое масштабирование – можно запускать несколько реплик Nginx без нагрузки Node.
- **Минусы**:
  - Если в проекте есть сервер‑сайд рендеринг или API, они не будут работать в этом образе; их нужно вынести в отдельный сервис.

## Вывод
Для текущего проекта, где Vue‑приложение полностью статическое, такой подход идеален. Компоненты сайта работают как обычные клиентские скрипты, а Nginx лишь обслуживает их. Если понадобится сервер‑сайд логика, её следует реализовать в отдельном контейнере.
